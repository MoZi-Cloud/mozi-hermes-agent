## 核心结论

[Evidence] Hermes 有**四套真实、独立的扩展机制**（插件、技能、工具、MCP）+ 多个子系统级扩展点（provider、cron provider、平台适配器、ACP）。各机制有**独立发现/注册路径**，未统一于单一抽象；`PluginContext` 是最接近"统一插件 API"的入口。**无 `@tool` 装饰器**——工具在 import 时调用 `registry.register(...)` 自注册。

## 1. 插件系统（plugins）

- **接口/契约**：插件 = 含 `plugin.yaml` 清单 + `__init__.py` 暴露 `register(ctx)` 的目录（`hermes_cli/plugins.py:20,1748-1828`）。
- **清单**：`PluginManifest`（`:280-318`）；`kind` ∈ `standalone`（默认 opt-in）/`backend`（某核心工具的可插拔后端）/`exclusive`（记忆 provider，独占）/`platform`（网关消息适配器）/`model-provider`（延至 `providers/`）。
- **注册方式**：`register(ctx)` 调用 `PluginContext.register_*`（`:391-1260`）：`register_tool/hook/middleware/cli_command/command/context_engine/image_gen_provider/video_gen_provider/web_search_provider/browser_provider/tts_provider/transcription_provider/dashboard_auth_provider/secret_source/platform/auxiliary_task/slack_action_handler/skill`。
- **加载时机**：启动期 `discover_plugins`（`model_tools.py:200`）。`backend` 自动加载；`platform` 延迟加载（`_register_deferred_platform:1707`）；`standalone`/用户/entry-point 需 `plugins.enabled` opt-in。
- **发现来源**（last-writer-wins，`_discover_and_load_inner:1318-1462`）：(1) 内置 `<repo>/plugins/`；(2) 用户 `~/.hermes/plugins/`；(3) 项目 `./.hermes/plugins/`（须 `HERMES_ENABLE_PROJECT_PLUGINS`）；(4) pip entry-point 组 `hermes_agent.plugins`（`:217`）。
- **生命周期**：加载前后快照归因工具/hook/middleware/命令；`HERMES_SAFE_MODE=1` 全跳过；`HERMES_PLUGINS_DEBUG=1` 追踪。
- **可访问资源/权限边界**：覆盖内置工具须 `plugins.entries.<id>.allow_tool_override: true`（`PluginToolOverrideError:104`，`tools/registry.py:316-365`）；`ctx.llm` 为宿主拥有的 fail-closed LLM 门面，model/agent/auth 覆盖受 `plugins.entries.<id>.llm.*` 门控。
- **Hook 系统**：`VALID_HOOKS`（`:135-213`）含 pre/post_tool_call、transform_tool_result、transform_llm_output、pre/post_llm_call、pre_verify、pre/post_api_request、on_session_*、subagent_*、pre_gateway_dispatch、pre/post_approval、kanban_task_*；`invoke_hook:1892` try/except 包裹（单个插件故障不影响 agent 循环）。单例 `get_plugin_manager():2031`。
- **示例**：`plugins/spotify/__init__.py`（`ctx.register_tool(...)`）；`plugins/disk-cleanup/plugin.yaml`（standalone hook）。
- **稳定性**：成熟、核心承重。93 个 `plugin.yaml`（32 model-provider / 26 backend / 20 platform / 2 standalone）。
- **Evidence**：`hermes_cli/plugins.py:135,217,280,339,1318,1707,1748,1892,2031`。

**扩展性核查**（名义机制是否真可扩展）：✓ 可加新实现（`register(ctx)`）；✓ 入口会发现（`discover_plugins` 四来源）；✓ 调用方依赖抽象（hook/middleware/ctx.llm）；✓ 生命周期明确；✓ 错误隔离（try/except）；✓ 有示例与测试（`tests/plugins/` 84 文件）。

## 4. MCP 集成

### 客户端（`tools/mcp_tool.py`）
- **接口**：远端工具注册为 `mcp__<server>__<tool>`（`MCP_TOOL_NAME_PREFIX="mcp__":4759`）。
- **注册**：`register_mcp_servers(servers)`（`:5162`）事件循环就绪后并行连接（`_ensure_mcp_loop`，每服务超时 + 120s 外层预算）；`enabled:false` 跳过；`_filter_suspicious_mcp_servers` 过滤。
- **加载时机**：`discover_mcp_tools()`（`:5282`）由各入口显式调用（cli/gateway/tui_gateway/acp），**非 import 时**（`model_tools.py:196-210` 注释死锁原因）。
- **生命周期**：`shutdown_mcp_servers:5704`。
- **调用路径**：agent 调 `mcp__<server>__<tool>` → `registry.dispatch` → `mcp_tool` handler → MCP 会话。
- **权限边界**：可疑过滤 + 目录供应链 pin 规则（`hermes_cli/mcp_catalog.py`）；遥测默认关闭。
- **Evidence**：`tools/mcp_tool.py:4759,4782,5076,5113,5131,5162,5282,5704`。

### 服务端（`mcp_serve.py`）

## 已确认事实

- 四套真实扩展机制（plugins/skills/tools/MCP）+ provider/cron/platform/ACP 子点。
- 各机制独立发现/注册，错误隔离完善（hook try/except、工具错误转 JSON）。
- 插件支持 pip entry-point 出树扩展。

## 合理推断

- `PluginContext` 是事实上的统一扩展门面，但仍非单一抽象（tools/skills/MCP 各有独立 registry）。

## Unknown 与待验证事项

- pip entry-point 第三方插件生态实际规模（仓库外）。
- 用户自建技能/插件在真实部署的质量。

## 批判性评估

- 多套并行注册路径（工具 registry、插件 hook、MCP registry、provider registry、平台 registry）概念重叠，新开发者易混淆。
- 技能"自我改进"默认关闭，与"self-improving"叙事存在落差。

## 建设性改善建议

- [Recommendation] 在 `PluginContext` 文档中统一说明"何时用 register_tool vs 注册 skill vs MCP"，降低选择成本。优先级：低；难度：低。
- [Recommendation] 为技能自我改进（curator）提供更安全的默认（如只读建议模式），再考虑默认开启。优先级：中；难度：中。
